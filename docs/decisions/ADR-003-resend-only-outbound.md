# ADR-003 — Resend as the only outbound mail channel, GoDaddy SMTP as fallback

- **Status:** Accepted 2026-06-13
- **Driver:** CTH platform (vendor blasts, confirmation mail, inbox replies, 2026 festival cycle).
- **Affected surfaces:** `lib/email.ts`, `lib/email-resend.ts`, `lib/email-smtp.ts`, `app/api/cron/notify/*`, `app/api/exhibitor/contract/sign`, `mail_messages.outbound_provider`.

## Context

The platform sends three classes of outbound mail:

1. **Vendor blasts** (verification, delay notice, broadcast). Bulk
   sends across hundreds of vendors. The 1 Jun 2026 verification
   blast hit the GoDaddy daily-cap mid-flight; 180 of 415 went and
   the remaining 235 had to be resumed by a launchd timer at 04:15
   Dubai. See `project_cth_verification_blast.md` in user memory.

2. **Confirmation mail** (application received, vendor approved,
   contract signed, ticket information). Single-recipient,
   transactional, triggered by user action.

3. **Operator replies from the unified inbox** (Sam answering a
   vendor mail from `app/(admin)/admin/bot-inbox/`). Single-recipient,
   threaded into an existing mail conversation, must carry the right
   `In-Reply-To` and `References` headers.

The Sam R10 incident (the dropped vendor approval email that
triggered the Phase 0 doctrine sweep, see ADR-0001) traced to the
GoDaddy SMTP fallback path missing `maxMessages: 20` for connection
recycling, which is now Law 5 in CTH-DOCTRINE. The same incident
surfaced that GoDaddy's daily send cap and throttle behavior are
operationally hostile to anything beyond low-volume confirmations.

Resend was introduced as the outbound channel for the verification
blast specifically because it does not have GoDaddy's throttle
shape. Resend's daily cap on the paid plan exceeds the festival's
total vendor pool, and the API is async-friendly.

The question is whether to dual-track Resend and SMTP for different
classes of mail (transactional via SMTP, bulk via Resend), or to
make Resend the only first-class outbound and treat SMTP as a
fallback only.

Deliverability also matters. Vendor-facing mail must arrive in the
inbox, not the spam folder. SPF / DKIM / DMARC alignment on
`cthalaal.co.za` is set up for both GoDaddy and Resend, but Resend's
infrastructure gives a stronger reputation signal at the volumes the
project sends.

Replyability also matters. When Sam sends from the inbox or when a
vendor receives a confirmation, the reply must land back in
`vendors@cthalaal.co.za` (the IMAP mailbox the unified inbox polls)
or in a Resend-managed mailbox that the IMAP poller can also read.
Today the IMAP mailbox is GoDaddy. Replies to Resend-sent mail must
carry a `Reply-To: vendors@cthalaal.co.za` header so the reply
threads back into the inbox.

CAN-SPAM and POPIA (South Africa's privacy law) both require an
unsubscribe mechanism on bulk vendor mail. Transactional confirmation
mail is exempt but bulk vendor blasts are not.

## Decision

**Resend is the only first-class outbound mail channel. GoDaddy SMTP
is a fallback path that runs only when Resend is unreachable or has
exhausted its rate budget for the day. Every outbound mail carries a
`Reply-To: vendors@cthalaal.co.za` header, a stable `Message-ID`, and
(for bulk classes only) a `List-Unsubscribe` header.**

### Channel routing rules

- Vendor blasts: Resend, batched at the Resend SDK's preferred batch
  size, idempotent on `vendor_application_id + batch_id`. SMTP is
  never used for blasts.
- Confirmation mail: Resend. SMTP fallback only if Resend send
  throws (network error, 5xx, rate limit). The fallback path applies
  `maxMessages: 20` per Law 5.
- Inbox replies: Resend. Reply carries the original message's
  `Message-ID` in `In-Reply-To` and prepends to `References` so the
  vendor's mail client threads it correctly. SMTP fallback applies
  the same threading headers.

### Headers required on every outbound

- `From: Cape Town Halaal <vendors@cthalaal.co.za>` (or campaign-
  specific sender for blasts, e.g. `Cape Town Halaal Festival
  <festival@cthalaal.co.za>`).
- `Reply-To: vendors@cthalaal.co.za`. This is non-negotiable. Without
  it, replies to a Resend-sent mail land in Resend's bounce path,
  not in the IMAP mailbox the inbox polls.
- `Message-ID: <uuid@cthalaal.co.za>`. Generated by the sender, not
  by Resend. Persisted to `mail_messages.message_id` for the
  outbound row. This is what step 2 of the name resolution chain in
  ADR-002 keys on.
- For bulk blasts only:
  - `List-Unsubscribe: <mailto:vendors@cthalaal.co.za?subject=unsubscribe>, <https://cthalaal.co.za/unsubscribe?token=...>`.
  - `List-Unsubscribe-Post: List-Unsubscribe=One-Click` per RFC 8058
    so Gmail's one-click unsubscribe works.

### Provider selection logic

`lib/email.ts` exposes a single `sendMail()` function. Internally it
calls `lib/email-resend.ts` first. On any of these conditions, it
falls through to `lib/email-smtp.ts`:

- Resend SDK throws a network error.
- Resend returns a 5xx.
- Resend returns a 429 (rate-limited) and the retry budget for the
  current send is exhausted.
- The `RESEND_API_KEY` env var is unset (development / disaster
  recovery).

The `mail_messages.outbound_provider` column records which path
actually fired (`resend` or `smtp`). The watchdog cron alerts if the
24-hour SMTP rate climbs above a small fraction of total outbound.

### List-Unsubscribe + suppression list

The unsubscribe link on bulk mail resolves to
`/unsubscribe?token=<hmac>` and inserts a row into a `mail_suppressions`
table on confirm. The blast sender filters against this table before
each batch. The mailto unsubscribe is monitored by Sam in the inbox;
manual confirmation flows through the same suppression table.

## Consequences

### Positive

- **One outbound channel for normal operation means one set of
  deliverability properties to tune.** Reputation, throttle shape,
  bounce handling all live with Resend.
- **The Sam R10 class of incident becomes structural.** GoDaddy SMTP
  is not on the hot path for confirmation mail anymore; if Resend is
  healthy, vendors get confirmations. SMTP only runs as a backstop,
  and Law 5's `maxMessages: 20` still applies on that path.
- **Replyability is preserved across both channels.** The
  `Reply-To: vendors@cthalaal.co.za` header means every reply lands
  in the IMAP mailbox, and the unified inbox catches it via the
  threading match in ADR-002's resolver (step 2).
- **Bulk-mail compliance is one helper away.** The List-Unsubscribe
  headers plus the suppression table satisfy CAN-SPAM and POPIA
  for the vendor-blast class. Gmail / Yahoo bulk sender requirements
  (DKIM, DMARC, one-click unsubscribe) are met.
- **`mail_messages.outbound_provider` makes the audit explicit.**
  Anyone reading the table sees per-row which channel sent the mail.
  A spike in SMTP usage is visible without reading logs.
- **Threading headers (`Message-ID`, `In-Reply-To`, `References`)
  are set by the sender, not the provider.** Switching providers
  later does not break thread continuity in vendors' mail clients.

### Negative

- **Resend is a single point of failure for the bulk class.** If
  Resend has a multi-hour outage during a blast window, the blast
  pauses. Mitigated by: blasts are idempotent and resumable
  (the verification blast precedent already proves this); SMTP is
  not a real alternative for bulk anyway.
- **Generating `Message-ID` in-app instead of letting Resend
  generate one means one more thing to get right.** Mitigated by:
  the generator is a single utility (`uuid + @cthalaal.co.za`); it
  has no per-call variance.
- **The suppression list is the project's responsibility, not
  Resend's.** Mitigated by: Resend exposes a suppression API too,
  but we keep ours because POPIA-mandated unsubscribe must work
  even if the project later moves off Resend.
- **Some confirmation mail (e.g. tickets) flows through WooCommerce
  + FooEvents, not through this codebase at all.** Mitigated by:
  Law 3 (FooEvents-no-fork) preserves that as canonical; this ADR
  governs the mail this codebase sends.

## Alternatives Considered

### A. GoDaddy SMTP as the primary, Resend as fallback (rejected)

Send confirmations and inbox replies through GoDaddy SMTP (the
mailbox that owns the address), use Resend only for blasts.

**Why rejected:**
- The Sam R10 incident is exactly this configuration. GoDaddy's
  throttle shape dropped a confirmation. Repeating the architecture
  repeats the failure mode.
- GoDaddy's daily cap is low enough that a single bad day (a vendor
  thread that goes 12 messages deep with attachments) can push the
  account into throttle and break unrelated confirmation sends.
- Deliverability reputation on GoDaddy's shared SMTP infrastructure
  is weaker than Resend's at the project's volume.

### B. Dual-track Resend + SMTP with class-based routing, both first-class (rejected)

Resend for blasts, SMTP for transactional, both treated as
first-class with no fallback relationship.

**Why rejected:**
- Two first-class channels mean two sets of headers, two reputation
  profiles, two throttle behaviors, two failure modes to monitor.
  The operational surface doubles for no clear gain.
- The Reply-To header alignment becomes ambiguous: who is the
  sender of record, GoDaddy or Resend? The vendor's mail client
  shows different things depending on which channel sent the
  thread's most recent message.
- Bounce handling diverges: Resend has a bounce webhook, SMTP has
  to be polled via the mailbox's NDR processing. Two bounce paths
  is two suppression-list paths.

### C. Amazon SES as the outbound provider (rejected)

Use SES instead of Resend.

**Why rejected:**
- SES requires an out-of-sandbox application and a slow-warmup
  reputation period that is operationally hostile for a six-month
  festival cycle.
- The SES SDK is heavier than the Resend SDK; the project is on
  Vercel and cold-start cost matters for cron-driven sends.
- Resend is already proven on this codebase (the verification blast
  succeeded after the GoDaddy cap hit); switching providers mid-
  cycle introduces risk without addressing a real defect.

### D. Postmark as the outbound provider (rejected)

Postmark has strong transactional deliverability.

**Why rejected:**
- Postmark splits transactional and broadcast across two separate
  account types and pricing models; the project would end up with
  two Postmark accounts to manage. This is the same operational
  surface problem as alternative B.
- Resend is sufficient for both classes at the project's volume
  and there is no engineering reason to switch.

## Reversibility

**High.** `lib/email.ts` is the single chokepoint. Replacing Resend
with another provider is one file's worth of change (a new
`lib/email-<provider>.ts` and a routing swap). The `Message-ID`
generation, threading header logic, and List-Unsubscribe construction
all live in app code and are provider-agnostic.

The `mail_messages.outbound_provider` column carries the historical
audit. Adding a new value (`postmark`, `ses`) is non-destructive.

The suppression list is owned by the project, not by Resend, so a
provider migration does not lose unsubscribe state.

If a class of mail later needs a different provider (for example,
a high-volume ticket-reminder blast that exceeds Resend's plan),
the routing function can pick a provider per mail class without
disturbing the unified inbox or the threading contract.
